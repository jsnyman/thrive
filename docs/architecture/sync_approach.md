# Sync Approach

This document describes the initial offline-first sync model based on an append-only event log.

## Storage Layout

Client

- Local SQLite queue store (`queued_event`) in OPFS.
- Local SQLite sync state store (`sync_state`) for cursor and last-sync metadata.

Server

- Canonical append-only event log in PostgreSQL.
- Server projections for reporting and cross-device reads via Postgres materialized views refreshed after each accepted event write.

## Sync Flow

1. Client appends a new event to its local queue store.
2. Client initiates sync via explicit user action (`Sync Now`) and after enqueueing core workflow actions in the current web flow (person create/edit, intake submit, sales submit, inventory status changes, and inventory adjustment requests).
3. Client pushes queued events in batches (max 100 events) with its last known server cursor.
4. Server validates/merges events, appends accepted events, and returns per-event ack.
5. Client acknowledges only `accepted`/`duplicate` queue entries.
6. Client pulls remote events since its cursor and advances cursor from `nextCursor`.
7. Client reads `GET /sync/status` for diagnostics and projection freshness metadata.

## Cursors and Idempotency

- Each event has a globally unique `eventId` generated by the client.
- The server treats `eventId` as idempotent and ignores duplicates.
- The server cursor is derived from `(recorded_at, event_id)` ordering and encoded/decoded by the API layer.

## Conflict Detection and Resolution

- Conflicts are detected when overlapping edits target the same entity in a way that violates business rules.
- The server records `conflict.detected` events that reference the conflicting events.
- Resolution is a administrator action recorded as `conflict.resolved` with a free-text reason.
- The system never drops original events; resolutions are additive.

## Failure Handling

- If a push fails, the client retains events and retries.
- If a pull fails, the client keeps its cursor and resumes later.
- If validation or merge checks fail, the server returns per-event rejection reasons and the client leaves unacked events queued for follow-up.

## Cadence

- Sync cadence is client-driven; current implementation is manual `Sync Now` plus action-triggered sync cycles after person, intake, sales, inventory, and procurement enqueue operations.
- After each accepted event write, the server refreshes materialized views and updates `projection_freshness` with refresh timestamp and projected cursor.
- Freshness metadata is available through `GET /sync/status` for operational monitoring and client-side sync diagnostics.
